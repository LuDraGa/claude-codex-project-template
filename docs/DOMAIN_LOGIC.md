# DOMAIN_LOGIC.md

## Purpose

Define financial invariants, idempotency behavior, and mutation failure policy.

## Core Invariants

1. Monetary unit is `credits_micros BIGINT` everywhere.
2. Credits are currency-agnostic internally; FX/currency concerns are isolated to topup package definition.
3. Ledger is append-only; no update/delete of committed rows.
4. Balance is derived (`SUM(delta_micros)`) from ledger rows.
5. Every mutation has an idempotency key scoped by `org_id`.
6. Same idempotency key + different payload is a conflict.
7. Redis lease can accelerate gating but cannot override ledger truth.

## Tenant and Identity Rules

- A user may belong to multiple orgs.
- Active org is selected from JWT claim `active_org_id`.
- All ledger, lease, outbox, and billing operations are scoped to the active org.

## Balance and Debt Rules

- Confirmed policy: debt disabled (`balance_micros` must remain `>= 0`).
- Debit path must reject if available lease + confirmed balance cannot satisfy request.
- Grace allowance during Redis+DB outage: disabled (strict fail-closed).

## Topup Package Rules

- Topups are package-based (not free-form amount entry).
- Minimum package amount is INR 100.
- Only predefined packages are accepted in v1.

## Currency Strategy

- Phase 1 settlement currency is INR only.
- One org wallet maps to exactly one settlement currency.
- Mixed-currency balances are disallowed.
- FX conversion is out of scope for phase 1.
- Future FX is applied only at checkout/package pricing layer, never inside ledger math.
- Multi-currency rollout should require package rows + gateway adapters, not ledger schema changes.

## Rate Locking

- Confirmed policy: `rate_id` is chosen at request start.
- All debits/refunds for the request carry the same `rate_id` unless an explicit step-level override policy is added.
- Rate config is immutable per `rate_id` version.
- Rate catalog source in v1 is static versioned config (move to DB-backed in phase 2).

## LLM Billing Lifecycle

1. Pre-call: apply lease-only debit in Redis for billable units.
2. `run_id` and `step_id` are generated by the gateway.
3. Post-call: compute actual usage credits and emit `UsageEvent` with stable idempotency key (`run_id:step_id`).
4. Event records include locked `rate_id`.
5. Worker applies canonical ledger debit from UsageEvent.
6. Token accounting source of truth is LiteLLM usage metrics only.

## Lease Lifecycle

- Lease key scope is `org_id` only.
- Lease TTL is 10 minutes.
- Low watermark for refill is 20% remaining lease.
- Refill triggers:
  - Successful topup completion.
  - Automatic refill when remaining lease <= 20% and ledger balance is sufficient.
- Refill uses deterministic ledger-backed calculation; lease never becomes the accounting authority.

## Tool Billing Lifecycle

1. MCP post-hook computes tool cost (fixed/per-tool/dynamic policy).
2. Emit `UsageEvent(kind=TOOL)`.
3. If tool fails after debit, emit full `REFUND` event tied to original `step_id`.

## Idempotency Rules

| Path | Key | Conflict Rule | Replay Behavior |
|---|---|---|---|
| LLM usage event | `run_id:step_id` | Different payload => conflict | Return stored outcome |
| Tool usage event | `run_id:step_id:tool_name` | Different payload => conflict | Return stored outcome |
| Ledger mutation | `(org_id, idempotency_key)` unique | Insert fails on key reuse | Existing row is authority |
| Cashfree webhook | `cf_payment_id` (plus org scope) | Same id + different payload/status => hard fail + incident | Ignore exact duplicate success |
| Lago event emit | `transaction_id=idempotency_key` | Lago dedupes by transaction | Safe retry |

## Failure Policy

- Redis unavailable + DB available:
  - Prefer DB-backed validation path if latency budget allows.
  - If DB fallback not feasible within SLA, fail closed.
- Redis unavailable + DB unavailable:
  - Fail closed (confirmed).
- Ledger write failure in worker:
  - Keep outbox row pending, exponential backoff retry.
- Lago failure after ledger success:
  - Do not rollback ledger; retry Lago emission from outbox state.
- Langfuse failure:
  - Non-blocking for financial commit; retry asynchronously.
- Cashfree signature verification failure:
  - Reject webhook and do not mint credits.

## Outbox Retry Policy

- Backoff sequence: 5s, 30s, 2m, 10m, 30m.
- Maximum attempts: 8.
- After 8 failed attempts, mark row `FAILED` and raise incident.

## Worker Concurrency Model

- Multiple consumers are allowed in v1.
- Consumers claim rows via `FOR UPDATE SKIP LOCKED`.
- Ordering is best effort by `next_retry_at`, with per-row idempotency preserving correctness.

## Reconciliation Rules

1. Ledger is canonical if downstream systems diverge.
2. Reconciliation cadence is hourly.
3. Every mismatch creates an incident record and corrective replay task.
4. Replays must preserve idempotency keys and never mutate historical rows.

## Error Taxonomy

- `DomainError`: invariant violation (insufficient credits, debt disallowed, idempotency conflict)
- `IntegrationError`: provider/network failures (Lago, Langfuse, Cashfree)
- `InfrastructureError`: data store or runtime issues (Redis, Postgres, worker runtime)
